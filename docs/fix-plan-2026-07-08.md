## Bug 1 (P0): Routing Stops Before All Providers Are Tried

### Problem

When three providers failed, the proxy returned an error to the client without
trying the remaining providers. The root cause was `stop_attempts=True` in
`scheduler_policy.classify_http_error` for non-fatal errors like `model_not_found`,
`schema_or_client_error`, and `not_retryable_status`. This caused the attempt loop
to `break` instead of `continue` to the next provider.

Additionally, `router.iter_attempts` capped `max_attempts` at the configured value
(default 6), which could be less than the total number of provider+key candidates,
causing premature exhaustion.

### Fix

1. **`scheduler_policy.py`**: Set `stop_attempts=False` for:
   - `model_not_found` (404/400 with model-not-found body) — another provider may
     support this model.
   - `schema_or_client_error` (422) — another provider may accept the schema.
   - `not_retryable_status` (unrecognized status codes) — another provider may
     handle this status differently.
   - `key_fatal_status` (401/403) — already `False`, confirmed correct.

2. **`router.py`**: Added `max_attempts = max(max_attempts, prov_count)` so
   `max_attempts` is always at least as large as the total candidate count.

3. **`sse2json.py`**: Simplified the "all attempts failed" error message returned
   to the client to `"All upstream providers are currently unavailable (req=..., Nms)"`
   instead of leaking `provider:status:error_type:reason` details. The detailed
   error list is still printed to server logs.

4. **`chat.py` / `responses.py`**: Applied the same error message simplification
   to these dead-code files (still listed in `pyproject.toml` py-modules) for
   consistency.

### Tests

- `tests/test_scheduler_policy.py`: Updated `test_model_not_found_*` and
  `test_422_*` to assert `stop_attempts=False`.

## Bug 2 (P1): New Provider Without Priority Defaults to Highest

### Problem

When adding a new provider via the Admin API or dashboard without an explicit
`priority` field, the provider received the default priority (0). Since
`priority_failover` sorts by descending priority and most existing providers
have positive priorities, a new provider with priority 0 would rank below
existing ones — but if existing providers also had 0 or negative priorities,
the new provider could unintentionally intercept traffic.

### Fix

**`config_manager.py`**: `add_provider` now auto-assigns a priority that is
lower than all existing provider priorities (`min(existing) - 1`) when no
explicit priority is set. If no other providers exist, it defaults to 0.

**`dashboard_src/src/app.js`**: The "Add Provider" modal form now omits
`priority` from the API payload when the field is 0, allowing the backend to
auto-assign. The input label shows `(auto if 0)`.

## Bug 3 (P1): Client Error Message Exposes Internal Provider Details

### Problem

The "all attempts failed" error returned to clients included a semicolon-joined
list of `provider:status:error_type:reason` strings, leaking internal provider
names, HTTP status codes, and error classifications to the API consumer.

### Fix

All three proxy handlers in `sse2json.py` (Chat Completions, Responses, Anthropic
Messages) and the dead-code files `chat.py` / `responses.py` now:
- Print the detailed error list to server logs via `print(flush=True)`.
- Return a simplified message to the client: `"All upstream providers are
  currently unavailable (req=..., Nms)"`.
- Record the detailed error string in observability via
  `OBSERVABILITY.record_request_end(..., error=detail_log)`.

## Bug 4 (P2): Frontend Priority Tooltip Says Wrong Direction

### Problem

`form.priority_tip` in `dashboard_src/src/i18n.js` said "lower number = higher
priority" in Chinese, but the system logic (and the English text after the fix)
uses "higher number = higher priority". The `router.py` sorts providers by
descending priority: `sorted(key=lambda entry: (-entry[1][2], ...))`.

### Fix

Updated the Chinese translation to: "数字越大 = 故障转移顺序中优先级越高（如 100
先于 90 先于 0）。新供应商默认为最低优先级。"

## Bug 5 (P2): Frontend pm.higher_first i18n Translation Is Inverted

### Problem

`pm.higher_first` in both `dashboard_src/src/i18n.js` and `dashboard/app.js`
had the Chinese translation "越小越优先" (smaller = higher priority), which is
the opposite of the actual system behavior and the English text "higher first".

### Fix

Changed the Chinese translation to "越大越优先" in both source and built files.

## Bug 6 (P2): Add Provider Form Lacks Clipboard Auto-fill

### Problem

Users had to manually copy and paste the base URL and API key into separate
fields when adding a new provider, which was tedious and error-prone.

### Fix

**`dashboard_src/src/app.js`**: Added:
- A "Paste & Auto-fill" button that reads the clipboard and detects URL/key
  patterns to auto-fill the `base_url` and `key` fields.
- Clipboard paste detection on the form that auto-fills fields when a pasted
  text contains both a URL and an API key.
- Preset provider chips (Anthropic, DeepSeek, Groq, OpenRouter, xAI, Mistral,
  SiliconFlow, Moonshot, Together) for one-click base URL and format filling.

**`dashboard/app.js`**: Same changes applied to the built file.
