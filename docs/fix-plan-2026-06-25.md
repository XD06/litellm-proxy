## Bug 1 (P0): Native Stream Pass-Through Never Switches Read Timeout

### Current Behavior

`OpenAIUpstreamClient.open_stream()` opens the upstream SSE connection with a
read timeout equal to `first_byte_timeout_s` (default 30s) whenever it is set:

```python
# upstream_client.py:367
read_timeout = first_byte_timeout_s if first_byte_timeout_s else timeout
urllib3_timeout = urllib3.Timeout(connect=connect_timeout, read=read_timeout)
```

All cross-format streaming adapters switch the socket read timeout from the
first-byte budget to the longer `read_timeout_s` (default 120s) as soon as the
first byte arrives:

```python
# stream_adapters.py:318-321
if not first_byte_received:
    first_byte_received = True
    if read_timeout_s:
        set_response_read_timeout(upstream, read_timeout_s)
```

But `relay_sse_stream()` — the native same-format pass-through path used for
Chat→Chat, Responses→Responses, and Anthropic→Anthropic streaming — has **no
`read_timeout_s` parameter and never calls `set_response_read_timeout`**:

```python
# stream_adapters.py:170-188
def relay_sse_stream(upstream, wfile, initial_lines=None, *, collect_usage=True):
    usage = empty_usage()
    for raw in initial_lines or []:
        wfile.write(raw); wfile.flush()
        ...
    for raw in upstream:
        wfile.write(raw); wfile.flush()
        ...
    return usage
```

The three call sites (`sse2json.py:2263`, `2557`, `2930`) also do not pass
`read_timeout_s`.

### Risk

This is the default code path. `native_stream_mode` defaults to `guarded`,
which skips prefetch and goes straight to `relay_sse_stream`. With
`first_token_timeout_s=30` (default), the socket read timeout stays at 30s for
the entire stream.

Any reasoning model that pauses longer than 30s between SSE events (DeepSeek R1
thinking, Claude extended thinking, slow generation after a long reasoning
phase) hits `socket.timeout`, the stream is interrupted, and the client sees a
broken response. The provider is healthy, but the proxy kills the connection.

This directly hurts the primary use case (reasoning-model streaming through
native-format providers).

### Fix Direction

1. Add `read_timeout_s: Optional[int] = None` parameter to `relay_sse_stream`.
2. After the first line written to the client, call
   `set_response_read_timeout(upstream, read_timeout_s)` once, mirroring the
   cross-format adapters.
3. Pass `read_t` from all three call sites in `sse2json.py`.
4. Keep the parameter optional so existing tests that call `relay_sse_stream`
   without it continue to work (no timeout switch when `None`).

### Acceptance Tests

- `relay_sse_stream` accepts `read_timeout_s` and calls
  `set_response_read_timeout` after the first line.
- Existing `relay_sse_stream` callers without `read_timeout_s` still work.
- A slow upstream that pauses 40s between events (with `first_token_timeout_s=30`
  and `read_timeout_s=120`) completes successfully instead of timing out.
- `python -m pytest tests/test_stream_adapters.py` passes.

## Bug 2 (P0): `open_stream` Error Path Leaks urllib3 Response

### Current Behavior

When the upstream returns HTTP >= 400, `open_stream` reads the error body,
raises `HTTPError`, but never closes the urllib3 response:

```python
# upstream_client.py:385-388
if resp.status >= 400:
    body_bytes = resp.read()
    fp = io.BytesIO(body_bytes)
    raise HTTPError(url, resp.status, getattr(resp, "reason", ""), resp.headers, fp)
```

The request was opened with `preload_content=False`, so the connection is not
returned to the pool automatically until garbage collection reaps the response
object. Under retry storms (multiple 429/5xx across keys/providers), many
half-consumed responses pile up before GC runs.

### Risk

Pool exhaustion under failure-heavy load. Each leaked response holds a
connection slot in the `PoolManager`. When `maxsize` is reached, new requests
block or fail. The effect is worse for providers with many keys that all return
errors, because the retry loop tries each key rapidly.

### Fix Direction

Wrap the status check so the response is always closed before raising:

```python
if resp.status >= 400:
    try:
        body_bytes = resp.read()
    finally:
        resp.close()
    fp = io.BytesIO(body_bytes)
    raise HTTPError(url, resp.status, getattr(resp, "reason", ""), resp.headers, fp)
```

### Acceptance Tests

- A 429 from upstream does not leave an open response; the connection is
  returned to the pool.
- `python -m pytest tests/test_upstream_client.py` passes.
- Existing retry behavior is unchanged (the same `HTTPError` is raised with the
  same body).

## Bug 3 (P1): Unconditional `print(flush=True)` On Every Request

### Current Behavior

`do_GET` and `do_POST` print one line per request with `flush=True`:

```python
# sse2json.py:2132
print(f"[proxy] GET {clean_path}", flush=True)
# sse2json.py:2754
print(f"[proxy] POST {clean_path}", flush=True)
```

`flush=True` forces `fflush(stdout)`. Under the threaded server with 20
workers, every request hits the C-level stdout mutex and the GIL for a
synchronous kernel write. This is a per-request global synchronization point
that scales poorly with concurrency.

### Risk

At high concurrency, the stdout lock becomes a bottleneck. Streaming requests
(which hold a worker for their full duration) compete with each other and with
fast non-streaming requests for the same lock. This adds latency jitter to the
hot path for no operational value — the same information is available in the
SQLite history and diagnostic logs.

### Fix Direction

Gate the per-request path log behind `observability.log_provider_on_each_request`
(the same flag already used for per-attempt logs), and drop `flush=True` so
stdout is buffered:

```python
if log_each:
    print(f"[proxy] POST {clean_path}")
```

This keeps the line for debugging when explicitly enabled, and removes the
synchronization point in normal operation.

### Acceptance Tests

- By default, no `[proxy] GET/POST` line is printed for normal requests.
- With `log_provider_on_each_request=true`, the line is printed (buffered).
- `python -m pytest tests/test_http_route_dispatch.py` passes.

## Bug 4 (P1): `_use_urllib3` Transport Selection Via `sys.modules`

### Current Behavior

```python
# upstream_client.py:128-133
def _use_urllib3(self) -> bool:
    routing = self.cfg.get("routing", {}) or {}
    default_transport = "urllib" if "unittest" in sys.modules else "urllib3"
    transport = str(routing.get("transport", default_transport)).lower().strip()
    return transport == "urllib3"
```

The default transport is chosen by whether `unittest` has been imported. If any
dependency (a plugin, a SDK, a debugging hook) imports `unittest` at runtime in
a production process, the proxy silently switches to the slower `urllib`
transport — no log, no error. The only signal is degraded performance.

### Risk

Silent performance regression that is extremely hard to diagnose. The
`unittest` check was intended to keep tests on the mock-friendly urllib path,
but `sys.modules` is global process state that any code can pollute.

### Fix Direction

- Change the default to always be `urllib3`.
- Let tests opt into urllib explicitly via config: set `routing.transport =
  "urllib"` in test fixtures (the tests already construct `OpenAIUpstreamClient`
  with a config dict).
- Keep the explicit `routing.transport` config override for both production and
  tests.

### Acceptance Tests

- In production (no config override), `_use_urllib3()` returns `True` regardless
  of `sys.modules`.
- Tests that pass `{"routing": {"transport": "urllib"}}` use the urllib path.
- `python -m pytest tests/test_upstream_client.py` passes.

## Bug 5 (P2): `_pool_managers` Unbounded Growth

### Current Behavior

```python
# upstream_client.py:154-193
def _pool_manager_for(self, proxy_url=None):
    key = str(proxy_url).strip() if proxy_url else "direct"
    cached = self._pool_managers.get(key)
    if cached is not None:
        return cached
    ...
    self._pool_managers[key] = manager
    return manager
```

Each unique `proxy_url` creates a `urllib3.PoolManager(num_pools=N,
maxsize=N)` that is never evicted. Adding and removing providers with different
proxy URLs through the dashboard accumulates managers indefinitely. Each holds
its own connection pool and SSL context.

### Risk

Memory and file-descriptor growth over time in long-running processes that see
frequent config edits. Not a crash, but a slow leak.

### Fix Direction

Cap the cache with a simple LRU bound (e.g. 32 entries). When the cap is
reached, evict the least-recently-used entry and call `.clear()` on its manager
to release connections before discarding.

Use `collections.OrderedDict` for LRU ordering: move-to-end on access, popitem
on overflow.

### Acceptance Tests

- Adding 40 distinct proxy URLs does not create more than 32 live managers.
- Evicted managers have `.clear()` called.
- Repeated access to the same proxy URL reuses the cached manager.
- `python -m pytest tests/test_upstream_client.py` passes.

## Bonus Fix: Routing State Migration By Key Fingerprint

### Problem

When providers' key lists were edited (add/remove/reorder), `migrate_state_from`
matched old state to new config by **index position**. If a key was removed from
the middle of the list, all subsequent keys inherited the wrong cooldown / failure
count state.

### Fix

Changed `migrate_state_from`, `dump_state`, and `load_state` in `router.py` to use
a **value-based key fingerprint** (`SHA-256` hash, truncated to 12 hex chars as
`key_hint`) instead of positional index. State now follows the actual key value
across list reordering, insertion, and deletion.

### Acceptance

- `python -m pytest tests/test_runtime_config_migration.py` passes.
- Removing a key from the middle of a provider's key list does not corrupt the
  cooldown state of remaining keys.
- Full suite: 351 passed.
